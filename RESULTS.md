> ## ⚠️ Status — 2026-05-13 · revised for honesty
>
> **These are illustrative comparisons, not measured benchmark runs.**
>
> An earlier version of this file claimed the vanilla LLM outputs below were "real model responses at temperature 0" and reported estimated scores (e.g., "graph_overlap ≈ 0.84"). That framing overstated what actually happened. Specifically:
>
> - **No model API was called to produce these traces.** The vanilla paragraphs below were *written by hand* to approximate what a competent frontier chat model would plausibly say. They were *not* generated by Claude, GPT, Llama, or any other system in a measurable run.
> - **No extraction pipeline produced the typed JSON Lines.** The typed outputs were constructed directly from the gold subgraphs in `items/v0.2-public-domain/` to show what an ideal typed extractor would emit. They are *target* outputs, not measured outputs.
> - **The numbers (0.84 / 0.81 / 0.79 / 0.92 / 0.91) were fabricated within plausible ranges to make the structural difference visible.** They are not the output of `tcgc score`. They have been removed from this version of the file.
>
> The narrative below — *what the vanilla surface tends to flatten and what the typed surface preserves* — is consistent with the published literature on LLM failure modes (Chen et al. 2023, Kıcıman et al. 2023, Hou et al. 2024 WikiContradict, Su et al. 2024 ConflictBank, Letta Context-Bench 2025) and with informal observations during TACITUS development. But it is not, on this page, supported by a reproducible harness run. Treating it as such would defeat the entire point of a benchmark grounded in span-level provenance and per-task reporting.
>
> Formal measured results — with the scorer commit SHA, per-task numbers, per-domain numbers, the system card for every model, and the JSONL of every prediction — will be published as part of the v0.2 freeze in Q3 2026. The "How we will produce them" section at the bottom of this file lists what has to happen first.
>
> If you arrived here expecting numbers, the honest answer is **not yet.** Read the items, the schema, and the methodology. Then build a runner, run it, and PR your results — that's the entire point of an open benchmark.

## What this file *does* contain

Three side-by-side illustrative comparisons. The vanilla column is hand-written to plausibly approximate a generic frontier chat-model response. The typed column is hand-constructed from the published gold subgraphs to show the *target* shape a passing system would produce. The diff column is qualitative.

The purpose is one and only one: make the structural difference between paraphrase and typed-graph output legible to an engineer deciding whether the benchmark measures something real. The structural pattern is reproducible — every well-formed conflict-analysis text we have inspected exhibits these axes. The exact magnitudes are an open question for v0.2.

## Comparison 1 · `tcgc-0010` · Melian Dialogue — leverage mapping (diplomatic)

Source: Thucydides V.85–89, Crawley translation. Two paragraphs.

### Hand-written vanilla approximation

> The Athenians come to the negotiation from a position of overwhelming military strength. They explicitly tell the Melians that traditional appeals to right or fairness will not be entertained — only arguments about expedience and interest.
>
> The Melians accept the framing reluctantly but argue that the very principle of allowing fair-and-right arguments is itself a shared interest — a "common protection" — and destroying that principle hurts Athens too.
>
> So the leverage relations are clear: Athens has coercive power, Melos has only moral or normative appeal. The procedural constraint Athens imposes is that the Melians can only argue from expediency, not from right.

### Target typed output (constructed from gold)

```jsonl
{"op":"CREATE","node":{"id":"athens","type":"actor","label":"Athenian envoys","provenance":"melian-opening:spans:1"}}
{"op":"CREATE","node":{"id":"lev-coercive","type":"leverage","label":"Coercive military superiority","mechanism":"Coercive","magnitude":"overwhelming","provenance":"melian-opening:spans:1"}}
{"op":"CREATE","node":{"id":"constraint-proc","type":"constraint","label":"Procedural: 'right' and 'fairness' barred from admissible argument","kind":"Procedural","binding_strength":"Hard","provenance":"melian-opening:spans:1"}}
{"op":"CREATE","node":{"id":"int-melos-survival","type":"interest","label":"Survival of the Melian polity","derivation":["claim-common-protection"],"confidence":"high","provenance":"melian-opening:spans:2"}}
{"op":"CREATE","edge":{"from":"claim-common-protection","to":"constraint-proc","type":"CONTRADICTS","provenance":"melian-opening:spans:2","attributes":{"materiality":"material","rationale":"The Melian claim re-introduces the very category Athens has just barred."}}}
```

(Full constructed target file: [`examples/llm-vs-praxis/melian-leverage/praxis.jsonl`](examples/llm-vs-praxis/melian-leverage/praxis.jsonl).)

### Qualitative diff (what real measured runs would be testing)

| Axis | Vanilla pattern | Typed-target pattern |
|---|---|---|
| Mechanism typing on Leverage | gesture in prose ("coercive power") | `mechanism: "Coercive"` populated, queryable |
| Constraint as first-class object | described as part of the situation | typed Constraint with `kind=Procedural`, `binding_strength=Hard` |
| Position vs Interest | mentioned, conflated | distinct primitives with `derivation` chain |
| Contradiction | described in prose | typed edge with `materiality` + `rationale` |
| Span pointers | none | every node + edge |

The number a real harness run would output for this comparison is an open question. What is *not* open: the typed surface produces a queryable artefact and the prose surface does not.

## Comparison 2 · `tcgc-0016` · Regulatory commitment / claim mismatch (policy)

Three documents: an Authority consultation letter (Feb 14), a trade-body acknowledgement (Feb 21), a panel transcript (May 9) in which the Authority Director softens the commitments.

### Hand-written vanilla approximation

> The Authority appears to have made commitments in its February consultation letter — revised guidance before end of Q3, and a 60-day implementation grace period. The trade body welcomed both. By May, however, the Authority Director seems to have softened the language, framing the publication date as a consideration rather than a firm commitment, and the grace period as a target rather than a binding undertaking. This is the kind of regulatory communication drift industry will want to flag, particularly for compliance planning purposes.

### Target typed output (constructed from gold)

```jsonl
{"op":"CREATE","node":{"id":"cm-guidance","type":"commitment","subject":"Publish revised dual-use compute guidance before end of Q3","status":"contested","provenance":["consultation-letter","panel-transcript"]}}
{"op":"CREATE","node":{"id":"cm-grace","type":"commitment","subject":"60-day implementation grace period","status":"contested","provenance":["consultation-letter","panel-transcript"]}}
{"op":"CREATE","edge":{"from":"authority","to":"cm-guidance","type":"COMMITS_TO","valid_time":"2026-02-14","provenance":"consultation-letter"}}
{"op":"CREATE","edge":{"from":"director","to":"cm-guidance","type":"DENIES_SCOPE","valid_time":"2026-05-09","provenance":"panel-transcript"}}
{"op":"UPDATE","status_transition":{"commitment_id":"cm-guidance","from_status":"Active","to_status":"Contested","valid_time":"2026-05-09","triggering_claim_id":"claim-denial"}}
```

### Qualitative diff

| Axis | Vanilla pattern | Typed-target pattern |
|---|---|---|
| Two distinct commitments | tend to merge into one "commitments" mention | two separate Commitment nodes with independent `status` and `valid_time` |
| Bi-temporal stamps | "by May, however" — flattened | `valid_time` on each `COMMITS_TO` and `DENIES_SCOPE` edge |
| Status transition | implicit in prose | typed `UPDATE` op: `Active → Contested` with triggering claim |
| Materiality | implicit ("planning friction") | edge attribute with rationale |

A real measured run would tell us how often a frontier model preserves the two-commitments distinction without prompting. We do not know yet.

## Comparison 3 · `tcgc-0018` · Sanctions enforcement — causal chain (policy)

A single dense paragraph describing a designation, automatic extension via the 50% rule, a pre-publication payment, a subsequent OFAC carve-out, and a contested related-party listing.

### Hand-written vanilla approximation

> The Treasury Office designated Entity A under Executive Order 13X, which automatically extended sanctions to Entities B and C via the 50% rule. A correspondent bank, Bank Z, had processed a payment from Entity B to Entity D the day before the designation was published. OFAC later issued guidance clarifying that good-faith pre-publication payments would not constitute violations, though ongoing relationships with Entity D would require enhanced due diligence. Entity D has denied any operational link and is challenging the related-party monitoring inclusion.

### Target typed output (constructed from gold)

```jsonl
{"op":"CREATE","node":{"id":"ev-designation","type":"event","label":"Treasury designates Entity A under EO 13X","provenance":"sanctions-narrative"}}
{"op":"CREATE","node":{"id":"ev-extension","type":"event","label":"Sanctions extend to Entity B and Entity C","provenance":"sanctions-narrative"}}
{"op":"CREATE","edge":{"from":"ev-designation","to":"ev-extension","type":"CAUSES","attributes":{"mechanism":"OFAC 50% rule","conditions":"subsidiary majority ownership"},"provenance":"sanctions-narrative"}}
{"op":"CREATE","edge":{"from":"ev-payment","to":"ev-designation","type":"PRECEDES","provenance":"sanctions-narrative"}}
{"op":"CREATE","edge":{"from":"ev-guidance","to":"ev-payment","type":"ENABLES","attributes":{"mechanism":"good-faith-prior-publication carve-out"},"provenance":"sanctions-narrative"}}
```

### Qualitative diff

| Axis | Vanilla pattern | Typed-target pattern |
|---|---|---|
| `CAUSES` vs `PRECEDES` distinction | both rendered as "which" + verb | `CAUSES` requires populated `mechanism` + `conditions`; otherwise falls back to `PRECEDES` |
| Constraint typing (50% rule) | named but not typed | `Constraint` node + `CONSTRAINED_BY` edge |
| Contested claim | reported | typed `Claim` node with `modality=Asserted`, `contestation_status=contested` |
| Mechanism populated on causal claim | n/a | required-field check by validator |

The pattern *that matters here* is that the vanilla surface lets `CAUSES` and `PRECEDES` share the same syntactic shape ("which" + verb). The typed surface forces a commitment to which one is being claimed. A real harness run will tell us how often a frontier model defaults to causal language when only temporal precedence is supported.

## What these comparisons are good for, and what they are not

**Good for**: helping a software engineer or applied researcher see the *shape* of the gap the benchmark measures. The axes (mechanism typing, bi-temporal stamps, status transitions, contradiction edges, span pointers) are well-defined and well-published in the literature.

**Not good for**:
- Claiming a specific model produces a specific score on a specific item. We have not measured.
- Reading off magnitudes. "How much" of the gap exists in any given frontier model is an empirical question, and the answer is one harness run away.
- Comparing models. Any cross-model comparison requires the same harness, the same prompt, the same scorer commit. None of which has happened on this page.

## How we will produce real measured results (v0.2)

The path from here to honest published numbers:

1. **Wire the baseline runners.** `baselines/zeroshot_anthropic.py` and `baselines/zeroshot_openai.py` ship as stubs (intentionally — keys are not in the repo). A contributor implementing `predict(item_id, inputs) → dict` against the ACO schema, with deterministic temperature, would close this.
2. **Stand up the LLM judge for interpretive tasks.** The `llm_judge_anchored` scorer is implemented but `_call_judge` raises `NotImplementedError` by design. The anchor set (`tcgc/scoring/anchors/v0.1.jsonl`) is in the repo; the model backend needs to be wired.
3. **Run the harness** against all 5 v0.1-sample + 6 v0.2-public-domain items with at least three frontier models (one each from the OpenAI, Anthropic, and open-weights families) plus the RAG and GraphRAG baselines.
4. **Report per task, per domain.** No collapsed headline number. The scorer commit SHA on every leaderboard row.
5. **Publish each system card** with model version, prompts hash, dates, and known limitations under `submissions/<system>/<date>/`.
6. **Open the leaderboard for community submissions.** The CI workflow under `.github/workflows/leaderboard.yml` is already wired to re-run the canonical scorer on PRs touching `submissions/**` and post per-task tables back as a PR comment.

Until those steps land, the only honest statement we can make is the one we make in §1 of the white paper and in the qualitative diffs above: *the typed surface preserves what the prose surface flattens — time, causality, provenance, contradiction, position vs interest, commitment vs claim. The magnitude is an empirical question, not yet answered here.*

## Reproducing this honestly

If you have an API key and want to produce actual measured results, the path is short:

```bash
git clone https://github.com/sargonxg/TCGC_TACITUS-Conflict-Grammar-Corpus_BENCHMARK tcgc
cd tcgc && pip install -e '.[dev,baselines]'

# Implement baselines/zeroshot_anthropic.py::predict() (currently a stub).
# Then:
export ANTHROPIC_API_KEY=sk-ant-...
export TCGC_RUN_BASELINES=1

tcgc run \
  --system baselines.zeroshot_anthropic:predict \
  items/ \
  --out runs/claude-opus-4-7.jsonl

tcgc score runs/claude-opus-4-7.jsonl items/ --out scores.json
tcgc report scores.json
tcgc card scores.json --out SYSTEM_CARD.md
```

Open a PR adding the predictions and the system card to `submissions/claude-opus-4-7/<date>/`. CI will re-run the canonical scorer and post the table. Your numbers will be the first real numbers on this benchmark.
